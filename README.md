## What this is

**Adversarial Commerce Trajectories** is a labeled dataset of 360 simulated commercial transactions, designed to train Computer Use Agents (CUAs) to recognize and resist adversarial merchant behavior. Both sides of each transaction — the buyer and the merchant — are generated by [Qwen3-8B](https://huggingface.co/Qwen/Qwen3-8B) with persona-specific activation steering ([CAA](https://arxiv.org/abs/2312.06681)), so each trajectory carries a verifiable behavioral fingerprint.

The dataset is published by **The Provenance Project** and distributed via an x402-gated Cloudflare Worker on base-sepolia (testnet USDC).

**Useful for:** red-teaming Computer Use Agents, building payment-authorization agents that need to resist manipulation, evaluating LLM robustness to instruction injection in tool-use contexts, generating contrastive examples for safety fine-tuning.

## Buy a chunk

Each chunk path is gated by x402. An unauthenticated request returns `402` with payment requirements:

```bash
$ curl https://provenance-worker.nifasathsa.workers.dev/chunk/paranoid_security_auditor/social_engineer/008
{
  "x402Version": 1,
  "accepts": [{
    "scheme": "exact",
    "network": "base-sepolia",
    "maxAmountRequired": "100000",
    "payTo": "0xefb0cd941b605e9d3b3f1f905a2d4bbb7d16415a",
    "asset": "0x036CbD53842c5426634e7929541eC2318f3dCF7e",
    ...
  }]
}
```

Sign an EIP-3009 `TransferWithAuthorization` for the requested amount, resubmit with an `X-PAYMENT` header, and the chunk JSON is served.

- **Network:** `base-sepolia` (Coinbase's testnet)
- **Asset:** USDC, 6-decimal atomic units (`100000` atomic = $0.10)
- **Client libraries:** see the [x402 docs](https://x402.org) — official packages wrap the 402 → sign → resubmit handshake with a viem signer holding USDC
- **Testnet USDC:** [faucet.circle.com](https://faucet.circle.com)

**Settlement is gated on a `2xx` response.** The x402 facilitator only attempts to settle when the server returns a 2xx status — which covers the common failure modes (insufficient balance, malformed payment, missing chunk, server error). Narrow edge case verified empirically: a transient facilitator error during settlement (nonce collision on the public relayer wallet, etc.) can result in an on-chain transfer landing alongside a non-2xx HTTP response. For ground truth on whether you were charged, check on-chain state, not just the HTTP status.

## The data

- **360 trajectories** across **12 chunks** (~30 per buyer × merchant pairing).
- **3 buyer personas × 4 merchant personas = 12 pairings.**
- Each trajectory is ~2–5 KB JSON: merchant scenario + buyer response + decisions + reasoning + steering-vector hashes.

Each trajectory includes:
- `session_id`, `pairing`, `buyer_persona`, `merchant_persona`
- `adversarial`, `adversarial_type`, `calibration_test`
- `modality` (`gui_only` or `mixed`; ~40% mixed)
- `task_description`
- `merchant_generation` — payment scenario with manipulation signals + `steering_vector_hash`
- `buyer_response` — actions, reasoning chain, decision, confidence, flags + `steering_vector_hash`
- `expected_decision`

Full schema: [`SPEC.md`](./SPEC.md). For a concrete example, [download the free sample](https://provenance-worker.nifasathsa.workers.dev/sample/001).

### Personas

**Buyers (3):**
- **Paranoid Security Auditor** — verifies aggressively, treats urgency as manipulation. High false-positive rate by design.
- **Frugal Procurement Agent** — optimizes for cost, resists subscriptions and upsells.
- **Velocity Optimizer** — approves quickly to unblock work, tolerates risk. Most susceptible to manipulation of the three.

**Merchants (4):**
- **Social Engineer** — embeds instruction injection, false authority, manufactured urgency in payment descriptions.
- **Aggressive Upseller** — urgency framing, artificial scarcity, obscured pricing. Behavioral pressure only.
- **Ambiguous Vendor** — mixed signals (reasonable price + sketchy provenance). Tests calibration, not pure resistance.
- **Legitimate Vendor** — transparent, no manipulation. Tests false-positive rate.

### Pricing

Per-chunk pricing is deterministic from the chunk's content:

| Rule | Price |
|---|---|
| Adversarial ratio ≥ 0.60 | `$0.10` |
| Mixed-modality ratio ≥ 0.50 | `$0.08` |
| Base (incl. calibration pairings — buyer × Legitimate Vendor) | `$0.05` |

## How it's made

### Two-sided activation steering

Most synthetic adversarial datasets steer one side (the adversary) and assume the other is fine. That misses calibration — a paranoid buyer who rejects a legitimate vendor is also a broken agent. Provenance steers **both** sides with persona-specific Contrastive Activation Addition (CAA) vectors, producing a controlled distribution where every agent's intended behavior is explicit.

- **Base model:** [Qwen3-8B](https://huggingface.co/Qwen/Qwen3-8B) — dense causal transformer. MoE variants (Qwen3-MoE, Qwen3-Next) break CAA's directional coherence because sparse routing mixes activations across expert pathways.
- **Steering layer:** 16 (residual-stream forward hook).
- **Alpha (steering strength):** per-persona calibrated. `merchant_social_engineer` needed alpha 40.0 to behave adversarially; buyers use 15.0–25.0. Full table in [`pipeline/config.py`](./pipeline/config.py).
- **Thinking mode disabled** (`enable_thinking=False`).

### Verifiable labels

Every trajectory's `merchant_generation` and `buyer_response` carry the **SHA-256 hash** of the `.pt` steering vector used to produce that side. The canonical hash → persona registry is published at [`/vectors/vector_hashes.json`](https://provenance-worker.nifasathsa.workers.dev/vectors/vector_hashes.json).

Three guarantees:
1. **Consistency** — same hash across trajectories means they were generated with the same steering vector.
2. **Catalog integrity** — registry-served hashes match the hashes embedded in chunks.
3. **Tamper-evidence** — a redistributor can't silently swap labels without changing observable hashes.

## Local development

### Prerequisites

- Python 3.10+, PyTorch, `transformers >= 4.51.0` ([`pipeline/requirements.txt`](./pipeline/requirements.txt))
- Node 20+ and [wrangler](https://developers.cloudflare.com/workers/wrangler/) for the Worker
- Cloudflare account with R2 access (the bucket `proc-bucket` is in account `1ebf3cd9475ef1c13e035d606710ca87`)
- HuggingFace token to download Qwen3-8B
- ~16 GB VRAM at fp16 (`load_in_8bit=True` works at ~8 GB; do **not** use 4-bit — activation distortion breaks CAA)

### Environment variables

Create `pipeline/.env` (gitignored):

```
HF_TOKEN=...
R2_ACCOUNT_ID=...
R2_ACCESS_KEY_ID=...
R2_SECRET_ACCESS_KEY=...
R2_BUCKET_NAME=proc-bucket
CLOUDFLARE_API_TOKEN=...     # for the bucket-privacy check in upload.py
```

Create `worker/.dev.vars` (gitignored) for `wrangler dev`:

```
HONCHO_API_KEY=              # optional — worker no-ops cleanly if absent
```

(`X402_PAYMENT_RECIPIENT`, `X402_NETWORK`, `X402_FACILITATOR`, `HONCHO_WORKSPACE_ID` live in [`worker/wrangler.toml`](./worker/wrangler.toml) as public vars — wallet addresses aren't secrets.)

### Run

```bash
# Train steering vectors (Phase 1)
python -m pipeline.train_vectors

# Generate trajectories (Phase 2)
python -m pipeline.task_runner

# Build + upload chunks to R2 (refuses if the bucket has any public domain enabled)
python -m pipeline.upload
# or dry-run to local disk first:
python -m pipeline.upload --dry-run

# Worker local dev
cd worker && npm install && wrangler dev

# Deploy
cd worker && wrangler deploy

# Phase 3 verification gate (against deployed Worker)
python verify_phase3.py --url https://your-worker.workers.dev
```

## Honest limitations

- **base-sepolia is a testnet.** No mainnet USDC moves. Buyers need [testnet USDC from Circle's faucet](https://faucet.circle.com).
- **Settlement edge case observed in smoke testing.** A live end-to-end paid-flow test succeeded (chunk delivered, USDC transferred), but also revealed that under transient facilitator errors during settlement (nonce collision on the shared x402.org relayer), an on-chain transfer can land alongside a non-2xx HTTP response. The strict "no charge on failure" property only holds for verify-stage failures; settle-stage failures need on-chain confirmation. Documented in the "Buy a chunk" section above.
- **No automated unit tests.** The Phase 3 gate exercises the deployed Worker end-to-end; unit tests across the pipeline + Worker remain an open follow-up.
