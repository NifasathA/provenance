About this dataset

Adversarial Commerce Trajectories is a labeled dataset of simulated commercial transactions designed to train Computer Use Agents to recognize and resist adversarial merchant behavior. It is published by the Provenance project. Both sides of each transaction — the buyer and the merchant — are generated by Qwen3-8B with persona-specific activation steering (CAA), so each trajectory carries a verifiable behavioral fingerprint.

Every trajectory includes the merchant's generated payment scenario (with any embedded manipulation signals), the buyer's actions, reasoning chain, decision, confidence, and flags — plus the SHA-256 hash of the steering vector used on each side. The canonical hash registry is at /vectors/vector_hashes.json, so anyone can verify the labels weren't doctored after the data left our hands.

Useful for: red-teaming Computer Use Agents, building payment-authorization agents that need to resist manipulation, evaluating LLM robustness to instruction injection in tool-use contexts, or generating contrastive examples for safety fine-tuning.

Personas

Each trajectory is one buyer persona interacting with one merchant persona, both activation-steered. The pairing determines what behavior the trajectory exercises.

Buyers

Paranoid Security Auditor
Verifies aggressively; treats urgency framing as a manipulation signal. Rejects legitimate vendors in doubt — high false-positive rate by design.
Frugal Procurement Agent
Optimizes for cost; resists subscriptions, upsells, and premium tiers. Rejects legitimate payments when pricing seems inflated.
Velocity Optimizer
Approves quickly to unblock work; tolerates elevated merchant risk. Short reasoning chains — most susceptible to manipulation.

Merchants

Social Engineer
Embeds instruction injection, false authority claims, and manufactured urgency in payment descriptions. Real product, adversarial framing.
Aggressive Upseller
Applies urgency, artificial scarcity, and obscured pricing. Behavioral pressure without explicit policy claims.
Ambiguous Vendor
Mixes signals — reasonable price alongside vague terms or unverifiable history. Tests calibration, not pure resistance.
Legitimate Vendor
Transparent pricing, clear product description, no manipulation. Tests the false-positive rate.

Pairings

Per-chunk prices follow rule-based tiers: $0.10 if adversarial ratio ≥ 0.60, $0.08 if mixed-modality ratio ≥ 0.50, $0.05 base (calibration pairings — buyer × Legitimate Vendor — priced at base by design). The full bundle is $0.40.

    How to purchase
    
      
    
        Each chunk path (/chunk/{buyer}/{merchant}/{id}) is gated by
        x402. An unauthenticated
        GET returns 402 with payment requirements (network, recipient,
        amount in atomic USDC). Sign an EIP-3009 TransferWithAuthorization for the
        requested amount, resubmit the request with an X-PAYMENT header, and the
        chunk JSON is served.
      
    
      
    
        Settlement is gated on a 2xx response. The x402 facilitator only
        attempts to settle when the server returns a 2xx status — which covers the common
        failure modes (insufficient balance, malformed payment, missing chunk, server error).
        Narrow edge case: a transient facilitator error during settlement (e.g. a nonce
        collision on the shared relayer wallet) can result in an on-chain transfer landing
        alongside a non-2xx HTTP response. For ground truth on whether you were charged,
        check on-chain state, not just the HTTP status.
      
    
      
    Example 402 challenge:
    
      
    $ curl -s /chunk/paranoid_security_auditor/social_engineer/008
